## Critical Finding: SC-10 Data Availability Constraint

**This is the most important planning-time finding.**

The CONTEXT.md decision specifies joining `seeker_profiles(first_name, last_name, email)`. However, inspection of the live schema reveals:

1. `seeker_profiles` does NOT have `first_name`, `last_name`, or `email` columns. The schema at `001_initial_schema.sql:51-72` shows no name columns — only `region`, `years_experience`, `sector_pref`, and other profile data.

2. Name and email data lives in `seeker_contacts` (columns: `phone`, `email`). The admin RPCs (`023_admin_rpcs.sql:317`) join `seeker_contacts sc ON sc.seeker_id = sp.id` and use `sc.first_name`/`sc.last_name`. Note: neither `first_name` nor `last_name` appear in the disk migration for `seeker_contacts` — they may be live-only columns added via Studio without a migration file.

3. `seeker_contacts` has strict RLS: employers can only read it after paying a placement fee (`002_rls_policies.sql:123-139` — "employers view after placement fee" policy requires `pf.acknowledged_at IS NOT NULL`). A direct PostgREST join from `MarkFilledModal` would silently return NULL for name/email for most applicants.

4. `match_scores.total_score` (not `score`) is the column name — `001_initial_schema.sql:178`. The PostgREST embed syntax would be `match_scores(total_score)`.

**Resolution options for the planner:**
- **Option A (recommended):** Use a SECURITY DEFINER RPC `get_applicants_for_job(p_job_id uuid)` that bypasses the seeker_contacts RLS, reads name from `seeker_contacts` (or `auth.users.email` as fallback), and returns `[{id, seeker_id, status, display_name, match_score}]`. This follows the existing mark_job_filled RPC pattern.
- **Option B:** Keep the MarkFilledModal query as-is for `id, seeker_id, status` and fetch `match_scores` and auth email via a separate read (two round-trips). Acceptable since this is employer UX, not a performance-critical path.
- **Option C:** Accept that only `region`+`years_experience` fields (which employers CAN see via the "employers view open-to-work seekers" policy) are available, display those as the identifier instead of a name.

The planner MUST choose between these — the CONTEXT.md decision to join `seeker_profiles(first_name, last_name, email)` cannot be executed literally because those columns do not exist on `seeker_profiles`. The spirit of the decision (show a human-readable identifier + match score) is correct; the specific column path needs adjustment.

### SC-11: ChipSelector Single-Select Usage

The `ChipSelector` component already handles `mode="single"`: when a chip is clicked in single mode, `onChange([optionValue])` is called — always an array of exactly one element. The `min_salary` field in the schema is `int`, not `int[]`. The storage contract (lower bound integer) means:

```typescript
// Chip options definition
const SALARY_CHIP_OPTIONS: ChipOption[] = [
  { value: '50000', label: '$50–60k' },
  { value: '60000', label: '$60–70k' },
  { value: '70000', label: '$70–80k' },
  { value: '80000', label: '$80–90k' },
  { value: '90000', label: '$90–100k' },
  { value: '100000', label: '$100–110k' },
  { value: '110000', label: '$110–120k' },
  { value: '120000', label: '$120k+' },
]
```

The `ChipSelector.value` is `string[]`, but the schema field is `number | undefined`. The `Controller` onChange adapter must convert `string[] → number | undefined`:

```typescript
// In Controller render prop
onChange={(vals) => field.onChange(vals[0] ? Number(vals[0]) : undefined)}
value={field.value ? [String(field.value)] : []}
```

The Zod schema already has `min_salary: z.coerce.number().optional()` — this is compatible because the value stored is still a number.

### SC-13: RLS Initplan Rewrite Pattern

**Before:**
```sql
USING (auth.uid() = user_id)
```

**After (advisor-compliant):**
```sql
USING ((select auth.uid()) = user_id)
```

The `(select auth.uid())` subquery form prevents Postgres from re-evaluating `auth.uid()` per-row (the "initplan" optimization). For join predicates the pattern extends to `WHERE ep.user_id = (select auth.uid())`.

The migration must use `CREATE OR REPLACE POLICY` or `DROP POLICY + CREATE POLICY` — Postgres RLS policies cannot be `ALTER`ed in place (there is no `ALTER POLICY ... USING` syntax for changing the expression; only the name can be altered). The safe pattern is `DROP POLICY IF EXISTS "name" ON table; CREATE POLICY "name" ON table ...`.

The existing policies use `BEGIN; ... COMMIT;` wrapping (CLAUDE.md §2 — Studio strips outer transaction but it is harmless to include for disk-file documentation purposes). Apply via Studio SQL Editor. The migration filename should be `030_rls_initplan_performance.sql` (next in sequence after 029).

## Common Pitfalls

### Pitfall 1: seeker_profiles has no name columns
**What goes wrong:** Planner writes `seeker_profiles(first_name, last_name, email)` in PostgREST join syntax — query returns 400 column not found.
**Why it happens:** CONTEXT.md decision references these columns; they don't exist on `seeker_profiles` in the schema.
**How to avoid:** Use an RPC or join `seeker_contacts` (with awareness of RLS gating). The `auth.users` email is available via admin-role context only. Most practical: SECURITY DEFINER RPC that returns display names for a given job's applicants.
**Warning signs:** PostgREST 400 on the applicants query or silent NULL for all name values.

### Pitfall 2: match_scores column is total_score, not score
**What goes wrong:** PostgREST join `match_scores(score)` returns 400 column not found.
**Why it happens:** The schema at `001_initial_schema.sql:178` names it `total_score`.
**How to avoid:** Use `match_scores(total_score)` in the select string.

### Pitfall 3: ChipSelector stores string[], min_salary wants number
**What goes wrong:** Submitting the form stores a string or array instead of a number in `min_salary`.
**Why it happens:** ChipSelector always calls `onChange(string[])`. The Controller adapter must convert.
**How to avoid:** Wrap in Controller with explicit adapter: `onChange={(vals) => field.onChange(vals[0] ? Number(vals[0]) : undefined)}`.

### Pitfall 4: ALTER POLICY does not change USING expression
**What goes wrong:** Migration uses `ALTER POLICY "name" ON table USING (new_expr)` — this syntax doesn't exist in Postgres.
**Why it happens:** Postgres `ALTER POLICY` only supports renaming or changing `TO`, `USING`, `WITH CHECK` role assignments — not expression replacement in most forms.
**How to avoid:** Use `DROP POLICY IF EXISTS "name" ON table; CREATE POLICY "name" ON table USING (new_expr);`

### Pitfall 5: Studio applies migration without registry row
**What goes wrong:** Applying 030 via Studio does not insert a row in `supabase_migrations.schema_migrations` — `list_migrations` will not show it.
**Why it happens:** Studio SQL Editor bypasses the migration registry (CLAUDE.md §2, NAMING.md §"Why two conventions co-exist").
**How to avoid:** After Studio apply, verify runtime artefacts (pg_policies showing updated USING clauses), not `list_migrations`. Update NAMING.md lookup table with the Studio-vs-registry note for 030.

### Pitfall 6: seeker_contacts RLS blocks employer join
**What goes wrong:** MarkFilledModal `.select('..., seeker_contacts(email)')` returns null for all applicants because the employer hasn't paid a placement fee for any of them.
**Why it happens:** "employers view after placement fee" policy requires `pf.acknowledged_at IS NOT NULL` — which is almost never true at the time of marking filled.
**How to avoid:** Do not join `seeker_contacts` directly. Use a SECURITY DEFINER RPC or fall back to data that employers can already see (seeker_profiles fields, match score).

### Pitfall 7: Nav.tsx seekerLinks — role === 'admin' branch
**What goes wrong:** Adding to `seekerLinks` without noticing that `Nav.tsx:34` also renders `seekerLinks` for `role === null` authenticated sessions (the `role === 'employer' ? employerLinks : seekerLinks` expression uses seekerLinks as default). Admin role falls back to seekerLinks too, but admins use AdminLayout not DashboardLayout, so Nav.tsx is not rendered for admins. No impact.
**How to avoid:** Confirmed safe — admin users hit AdminGate at `/admin` and never render Nav.tsx.

### Pitfall 8: Sidebar.tsx `end` prop on sub-routes
**What goes wrong:** The new My Documents link renders as active when navigating to `/dashboard/seeker/documents/anything` — or conversely, never active because `end={item.to.includes('/dashboard')}` is true and the exact match fails.
**Why it happens:** Sidebar.tsx line 52 uses `end={item.to.includes('/dashboard')}`. `/dashboard/seeker/documents` includes `/dashboard`, so `end=true` — this means the link is only active on the exact path, which is correct behaviour for a leaf route.
**How to avoid:** No action needed — the `end` prop logic is already correct for this route.
