## Decisions Made

- DROP POLICY IF EXISTS + CREATE POLICY (not ALTER POLICY) — ALTER POLICY cannot change USING expressions in Postgres; DROP/CREATE is the only path
- Tables covered include all sources of `auth.uid()` found across migrations 002, 005, 009, 015, 019, 020, 024 — comprehensive sweep not just 002
- Instrumentation in `loadRole` only (inner DB query), not `loadRoleWithTimeout` (outer race wrapper) — measures pure DB round-trip to isolate whether latency is at DB layer or Lock/promise layer
- Migration 031 applied via Studio (per CLAUDE.md §2) — MCP is `--read-only` by default; Studio is the preferred path for one-off DB writes
- Postgres normalization documented: `(select auth.uid())` is stored as `( SELECT auth.uid() AS uid)` in pg_policies.qual; LIKE pattern for future audits must use `'% SELECT auth.uid() AS uid%'` not `'%(select auth.uid())%'`

## Deviations from Plan

None to code — plan executed exactly as written. Policy names and expressions were read from source migration files before writing 031, as the plan required. ## Issues Encountered

**Postgres normalization of RLS USING clause:** The plan's verification query `LIKE '%(select auth.uid())%'` did not match stored pg_policies.qual because Postgres normalises `(select auth.uid())` to `( SELECT auth.uid() AS uid)` on storage. The operator identified this during Task 3 and provided the corrected query `LIKE '% SELECT auth.uid() AS uid%'`. Actual verification result: 34 compliant policies, 0 bare. Migration 031 applied correctly — the discrepancy was only in the verification query, not the migration body itself. Documented in NAMING.md for future RLS auditors.
