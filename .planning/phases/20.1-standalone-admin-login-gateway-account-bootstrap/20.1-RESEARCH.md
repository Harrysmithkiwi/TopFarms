### Pattern 1: Hybrid AdminGate (CRITICAL)

**What:** A wrapper component co-located with `AdminLoginPage` that branches on auth+role state. Replaces the existing `ProtectedRoute requiredRole="admin"` wrapper at the `/admin` index route only.

**When to use:** Only at `/admin` index. Sub-routes (`/admin/employers`, etc.) keep `ProtectedRoute` because they should redirect non-admins; the gateway is the only inverse-redirect (render LoginPage instead of redirecting to `/login`).

**Branching logic** (mirrors `ProtectedRoute.tsx` decision tree, but inverted at the unauthenticated branch):

```tsx
// src/pages/admin/AdminLoginPage.tsx
export function AdminGate() {
  const { session, role, loading } = useAuth()

  // 1. Initial load — same JSX as ProtectedRoute.tsx:13-26 (mirror byte-for-byte)
  if (loading) {
    return <SpinnerBlock />  // duplicated inline, not extracted
  }

  // 2. Unauthenticated — render login form (this is the inversion vs ProtectedRoute)
  if (!session) {
    return <AdminLoginPage />
  }

  // 3. Authenticated but role still resolving (AUTH-FIX-02 race) — same JSX as ProtectedRoute.tsx:41-55
  if (role === null) {
    return <SpinnerBlock />  // duplicated inline
  }

  // 4. Authenticated non-admin — inline access-denied (NOT redirect, per CONTEXT Q3)
  if (role !== 'admin') {
    return <AccessDeniedView role={role} />
  }

  // 5. Admin — render AdminLayout + DailyBriefing
  return <AdminLayout><DailyBriefing /></AdminLayout>
}
```

**main.tsx change** (single route entry, lines 212-221):

```tsx
// BEFORE (verbatim from src/main.tsx:212-221):
{
  path: '/admin',
  element: (
    <ProtectedRoute requiredRole="admin">
      <AdminLayout>
        <DailyBriefing />
      </AdminLayout>
    </ProtectedRoute>
  ),
},

// AFTER:
{
  path: '/admin',
  element: <AdminGate />,
},
```

**No nested route children change.** All 4 sub-routes at lines 222-261 (`/admin/employers`, `/admin/seekers`, `/admin/jobs`, `/admin/placements`) preserve their existing `<ProtectedRoute requiredRole="admin"><AdminLayout>...</AdminLayout></ProtectedRoute>` wrapping verbatim. The `createBrowserRouter` route array uses sibling entries (NOT nested `children`), so removing/changing one route doesn't cascade.

## Common Pitfalls

### Pitfall 1: AUTH-FIX-02 race window (`session` set, `role === null`)

**What goes wrong:** AdminGate could flash login form to an admin user during the role-loading 3s window.

**Why it happens:** `AuthContext.tsx:81-86` calls `loadRoleWithTimeout(initialSession.user.id)` AFTER `setSession(initialSession)`. There's a window where `session` is truthy but `role` is `null`. `loading` flips to `false` either when the timeout fires (3s) OR when `loadRole` resolves.

**Critical sequence after `signIn()` resolves** (verified from `AuthContext.tsx:96-122`):
1. `supabase.auth.signInWithPassword` resolves → returns success
2. Supabase fires `SIGNED_IN` event on the auth listener
3. AuthContext callback runs: `setSession(newSession)` (synchronous)
4. AuthContext awaits `loadRoleWithTimeout(newSession.user.id)` (up to 3s)
5. On success: `setRole(result.role)` — admin's role populates here
6. After listener returns: `setLoading(false)` (only if `event !== 'INITIAL_SESSION'`)

There's a race where between steps 3 and 5, AdminGate could see `{ session: truthy, role: null, loading: ??? }`.

**How to avoid:** Mirror `ProtectedRoute.tsx:41-55` BYTE-FOR-BYTE. Check `if (role === null) return <Spinner>` BEFORE checking `if (role !== 'admin')`. This catches both:
- Hard refresh: `loading: true` → spinner (covered by step 1 of branch logic)
- AUTH-FIX-02 race: `loading: false, session: truthy, role: null` → spinner (covered by step 3)

**Warning signs:** Test mock returns `{ session: truthy, role: null, loading: false }` and AdminGate flashes `<AccessDeniedView>` — bug. Should render spinner.

### Pitfall 2: Lockout — operator removes admin role from old account before new account UAT passes

**What goes wrong:** Harry runs Studio SQL to remove admin from `harry.symmans.smith@gmail.com` BEFORE confirming `admin@topfarms.co.nz` works end-to-end. If the new account has a typo / wrong password / mailbox unreachable, Harry is now locked out of admin entirely. To recover: Studio SQL re-grant — but that requires Studio access, which is fine, but adds friction during a marketing window.

**Why it happens:** Sequencing error. Steps 1-5 of the bootstrap plan must run in strict order.

**How to avoid:** Plan 5 of Phase 20.1 MUST sequence operator steps as:
1. Studio Auth → Add user `admin@topfarms.co.nz` + password
2. Studio SQL → INSERT/UPDATE `user_roles` row for new user → `role='admin'`
3. Verify via read-only MCP: `SELECT u.email, ur.role FROM auth.users u JOIN user_roles ur ON ur.user_id=u.id WHERE u.email='admin@topfarms.co.nz'`
4. **UAT: sign out, sign in via standalone `/admin` AdminLoginPage with new account** → confirm AdminLayout renders, drawer works
5. ONLY AFTER step 4 PASS: Studio SQL → UPDATE `user_roles` row for old account → `role='seeker'`
6. Re-verify: sign in with old account → `/admin` shows AccessDenied

If step 4 fails, fix the new account before touching the old role. Document this rollback explicitly in the plan.

**Warning signs:** Plan does step 5 before step 4, or bundles 4+5 in a single operator action.

### Pitfall 3: StatusBanner has no `'error'` variant

**What goes wrong:** Planner writes `<StatusBanner variant="error">Access denied...</StatusBanner>` and TypeScript rejects: variant union is `'shortlisted' | 'interview' | 'offer' | 'declined'`. There is no `'error'`. Worse, the `'declined'` variant has hardcoded title copy ("Unfortunately, this application wasn't successful.") tied to applicant context.

**Why it happens:** Phase 19 `StatusBanner` was scoped to applicant-status surfaces; admin error UX wasn't anticipated.

**How to avoid:** Two options, both acceptable:
- **Option A (recommended for 20.1):** Use a small inline `role="alert"` div styled with `--color-danger` tokens (mirrors Phase 20-05 pattern for ProfileDrawer error display — see `STATE.md` decision: "ProfileDrawer error display uses inline `role='alert'` div with --color-danger tokens"). Per CONTEXT this is acceptable; CONTEXT calls out "StatusBanner" but the planner has discretion on copy/variant.
- **Option B:** Add a 5th StatusBanner variant `'error'` with custom title prop. Adds API surface to a v2 primitive — mild scope creep.

CONTEXT specifically allows planner discretion on this; recommend Option A to keep the surface area minimal.

**Warning signs:** TypeScript error on `variant="error"` after writing the access-denied JSX.

### Pitfall 5: Sidebar restructure — preserve existing rendering on tablet/mobile

**What goes wrong:** Planner adds the footer slot but breaks the existing `hidden md:flex` responsive behavior — sidebar disappears on desktop, or footer overflows on small screens.

**Why it happens:** Existing aside has classes `hidden md:flex flex-col w-60 flex-shrink-0 min-h-[calc(100vh-56px)] border-r`. The `flex flex-col` is already set, and `min-h` already gives space. Adding `mt-auto` to footer pushes it to the bottom of the available height.

**How to avoid:** Verbatim restructure (verified by reading current Sidebar.tsx):

```tsx

    {items.map(...)}
  </nav>
</aside>

// AFTER:
<aside className="hidden md:flex flex-col w-60 flex-shrink-0 min-h-[calc(100vh-56px)] border-r" ...>

    >
      <LogOut size={18} />
      <span>Sign Out</span>
    </button>
  </div>

```

`mt-auto` only works because `<aside>` is a flex column AND has a min-height — both already true. No restructure needed.

**AdminSidebar reference:** AdminSidebar uses TOP-positioned escape hatch ("Back to app" via `NavLink` to `/dashboard/...`) — NOT bottom-positioned. Operator note in CONTEXT confirms: consumer Sidebar uses BOTTOM Sign Out, AdminSidebar uses TOP Back-to-app. Different position, same "footer slot" mental model.

**Warning signs:** `mt-auto` doesn't push down → forgot the parent `flex flex-col` is set, or applied to wrong child.

### Pitfall 6: Test mock pattern — `vi.hoisted` for SUT-imported supabase

**What goes wrong:** Test imports `AdminLoginPage` (the SUT) which transitively imports `supabase` from `@/lib/supabase`. The test then tries `vi.mock('@/lib/supabase', () => ({ supabase: { auth: { signInWithPassword: mockFn } } }))` and gets a TDZ error because `mockFn` is initialized AFTER `vi.mock` is hoisted.

**Why it happens:** Phase 20-06 surfaced this exact issue (see `STATE.md` Phase 20-06 decisions). vi.mock factory bodies run BEFORE module code; any `const mockX = vi.fn()` declared at the top of the test file is undefined when the factory runs.

**How to avoid:** Use `vi.hoisted` (already used in `tests/select-role.test.tsx:26-28`):

```ts
const mocks = vi.hoisted(() => ({
  signIn: vi.fn().mockResolvedValue({ data: { session: null, user: null }, error: null }),
}))

vi.mock('@/hooks/useAuth', () => ({
  useAuth: vi.fn(() => ({
    session: null, role: null, loading: false,
    signIn: mocks.signIn,
    signOut: vi.fn(),
    // ...other AuthHookReturn fields
  })),
}))
```

For tests that mock `useAuth` directly (the simpler pattern from `admin-protected-route.test.tsx`), the `vi.mocked(useAuth).mockReturnValue({...})` pattern works without hoisted, because `useAuth` itself is the leaf mock — no transitive supabase import is needed.

**Warning signs:** Test fails with `Cannot access 'mockX' before initialization` or `mocks.signIn is not a function`.

### Pitfall 7: Helper file location — avoid circular imports

**What goes wrong:** Planner puts `dashboardPathFor` in `src/contexts/AuthContext.tsx` (since it's role-aware), then `Login.tsx` and `Nav.tsx` (which import `useAuth` from there) consume it — fine. But `ProtectedRoute.tsx` (which also imports `useAuth`) consumes it too, and any future addition to `routing.ts` that imports a router component might cycle.

**How to avoid:** Per CONTEXT Q4: put the helper at `src/lib/routing.ts` — pure function, zero imports. Cannot create a cycle. Avoids:
- `src/types/domain.ts` (mixes types with runtime)
- `src/contexts/AuthContext.tsx` (cycles via useAuth consumers)
- `src/lib/utils.ts` (currently only `cn()` — single-responsibility)

**Verified import diet:** `src/lib/routing.ts` imports only `import type { UserRole } from '@/types/domain'`. Type-only import, zero runtime deps.

### Pitfall 8: `--read-only` MCP flag for the bootstrap operator action (CLAUDE.md §2)

**What goes wrong:** Operator tries to run the INSERT/UPDATE bootstrap SQL via Supabase MCP (`apply_migration` or `execute_sql`) and gets blocked because MCP is `--read-only ON`. Or worse, flips MCP off, makes the change, doesn't flip back.

**How to avoid:** Per CLAUDE.md §2 — use Supabase Studio SQL Editor for one-off DB writes. The bootstrap SQL bodies in CONTEXT.md are copy-paste-ready into Studio. Verification queries (read-only) can run via MCP without flag-flipping.

**Documented sequence:**
1. Studio Auth → Add user (no SQL needed)
2. Studio SQL Editor → paste INSERT/UPDATE → run
3. MCP read-only → verification SELECT
4. UAT in browser
5. Studio SQL Editor → role removal UPDATE → run
6. MCP read-only → verification SELECT

No MCP flag-flip required. No restart needed.

### Pitfall 9: AdminLoginPage post-login navigation — let AuthContext drive

**What goes wrong:** AdminLoginPage's `onSubmit` adds a manual `navigate('/admin')` after `signIn()` resolves. This races with AuthContext's role-loading: navigate fires, ProtectedRoute on next render sees `{session: truthy, role: null}`, bounces to `/auth/select-role` mid-flight.

**Why it happens:** Login.tsx has the same trap and works around it via `didSubmit.current` ref + `useEffect` that waits for `!loading && session && role` (Login.tsx:46-51). AdminLoginPage shouldn't do this — it lives INSIDE the AdminGate at `/admin`, so once AuthContext populates state, AdminGate re-renders and the right branch (admin → AdminLayout, non-admin → AccessDenied) takes over.

**How to avoid:** AdminLoginPage's `onSubmit` should:
1. Call `signIn(email, password)`
2. On error: setError + render in StatusBanner / inline alert
3. On success: do nothing — AdminGate's re-render via AuthContext state-change handles it

This is simpler than Login.tsx because AdminLoginPage is rendered AT `/admin`, not at `/login` — no destination path is needed.

**Warning signs:** AdminLoginPage imports `useNavigate` from `react-router` — likely indicator of this trap.
