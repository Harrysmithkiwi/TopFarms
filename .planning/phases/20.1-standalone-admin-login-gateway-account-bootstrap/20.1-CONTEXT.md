<decisions>
## Implementation Decisions

### Architecture — `/admin` hybrid route (Q1)

- `/admin` no longer wraps in existing `ProtectedRoute`. The route renders a hybrid page that branches on auth+role state:
  - `loading` → spinner (mirror `ProtectedRoute.tsx:13-26` pattern inline; do NOT extract to shared primitive in this phase)
  - `!session` → `AdminLoginPage` (login form)
  - `session && !role && loading` → spinner (handle the AUTH-FIX-02 timeout race; mirror `ProtectedRoute.tsx:41-55`)
  - `session && role && role !== 'admin'` → "Access denied" inline state (NOT redirect)
  - `session && role === 'admin'` → `AdminLayout` + admin children
- The 5 admin sub-routes (`/admin/employers`, `/admin/seekers`, `/admin/jobs`, `/admin/placements`) keep their existing `ProtectedRoute requiredRole='admin'` wrapping. Only the `/admin` index route changes to hybrid.

### AdminLoginPage component (Q1, Q2, GA-1)

- New component at `src/pages/admin/AdminLoginPage.tsx`
- Email + password form ONLY. NO Google OAuth button. NO Facebook OAuth button.
- Composes Phase 19 v2 primitives: `Input` (`src/components/ui/Input.tsx`) + `Button` (`src/components/ui/Button.tsx`) + `StatusBanner` (`src/components/ui/StatusBanner.tsx`) for inline errors
- Uses existing `useAuth().signIn(email, password)` from `AuthContext.tsx`
- After successful sign-in: AuthContext `onAuthStateChange` populates session + role; the `/admin` hybrid wrapper re-renders and routes to AdminLayout when `role === 'admin'`, or "Access denied" otherwise
- Error states inline (bad credentials, network failure) — no toast; render in `<StatusBanner variant="error">`
- "Forgot password?" link below the form pointing to existing `/forgot-password` (GA-1)
- "Back to main site" link below the form pointing to `/` (operator's addition — escape hatch for accidental landings)
- Loading state on hard refresh: render the same spinner block as `ProtectedRoute.tsx:13-26` (Q2; inline duplication accepted; "extract to AuthLoadingSpinner" deferred)

### Access denied UX (Q3)

- Inline "Access denied" state inside the AdminLoginPage shell — NOT a redirect
- Composition: `<StatusBanner variant="error">Access denied — your account does not have admin privileges</StatusBanner>` + `<Link to={dashboardPathFor(role)}>Back to your dashboard</Link>`
- Reasoning: redirect to `/login` confuses (user IS authenticated); redirect to dashboard blurs the rejection. Inline is operationally cleanest.

### `dashboardPathFor` helper (Q4)

- New file: `src/lib/routing.ts`
- Signature: `export function dashboardPathFor(role: UserRole): string` where `UserRole = 'employer' | 'seeker' | 'admin'` (already defined in `src/types/domain.ts:1`)
- Body: `return role === 'admin' ? '/admin' : '/dashboard/${role}'`
- TDD: write `tests/dashboard-routing.test.ts` with 3 explicit unit tests (employer/seeker/admin) BEFORE swapping any callsite. Test file ships first.
- Refactor swaps 5 callsites in one atomic commit:
  1. `src/pages/auth/Login.tsx:48`
  2. `src/pages/auth/VerifyEmail.tsx:35`
  3. `src/components/layout/ProtectedRoute.tsx:63`
  4. `src/pages/auth/SelectRole.tsx:30`
  5. `src/components/layout/Nav.tsx:112`
- Existing `tests/admin-protected-route.test.tsx` provides regression coverage for ProtectedRoute callsite
- File location avoids: `src/types/domain.ts` (mixes types with runtime), `src/contexts/AuthContext.tsx` (cycles via useAuth consumers)

### Sign Out button in dashboard sidebar (GA-4)

- Modify `src/components/layout/Sidebar.tsx` to add a footer slot at the bottom of the aside
- Restructure: `<aside class="flex flex-col"><nav>{items}</nav><div class="mt-auto p-3 border-t">{signOutButton}</div></aside>`
- "Footer" means BOTTOM. Use `mt-auto` to push the Sign Out button down (existing aside has `min-h-[calc(100vh-56px)]` so there's space)
- Button calls `signOut()` from `useAuth()` (already exists at `AuthContext.tsx:179`)
- After sign-out, `onAuthStateChange` clears session → `ProtectedRoute` redirects to `/login` automatically. No explicit `navigate()` needed.
- Visible on both employer (`/dashboard/employer/*`) and seeker (`/dashboard/seeker/*`) routes since `Sidebar.tsx` is the shared component for both
- Role-agnostic — the button itself doesn't care what role the user is
- Test: `tests/sidebar-signout.test.tsx` — render Sidebar, click Sign Out, assert `signOut` mock called once

### Account bootstrap (GA-2)

- Method: Supabase Studio Authentication dashboard → Users → Add user → email + password (mirror Phase 20-08 Task 4 operator-action pattern)
- Email: `admin@topfarms.co.nz`
- Password: operator chooses, stores in password manager labelled `admin@topfarms.co.nz — TopFarms Phase 20.1`
- After Studio create-user: one-shot Studio SQL to assign admin role:
  ```sql
  INSERT INTO public.user_roles (user_id, role, is_active)
  VALUES (
    (SELECT id FROM auth.users WHERE email = 'admin@topfarms.co.nz'),
    'admin',
    true
  )
  ON CONFLICT (user_id) DO UPDATE SET role = 'admin', is_active = true;
  ```
- Verification query (read-only MCP):
  ```sql
  SELECT u.email, ur.role, ur.is_active
  FROM auth.users u
  JOIN public.user_roles ur ON ur.user_id = u.id
  WHERE u.email = 'admin@topfarms.co.nz';
  ```
  Expected: 1 row, `role='admin'`, `is_active=true`.

### Old admin role removal (GA-3)

- Phase doesn't close until BOTH conditions are true:
  1. `admin@topfarms.co.nz` UAT pass (sign in via standalone AdminLoginPage at `/admin` → AdminLayout renders → drawer smoke test)
  2. Admin role removed from `harry.symmans.smith@gmail.com` (Studio SQL):
     ```sql
     UPDATE public.user_roles
     SET role = 'seeker', is_active = true
     WHERE user_id = (SELECT id FROM auth.users WHERE email = 'harry.symmans.smith@gmail.com');
     ```
- Verification query: confirm Harry's account row is `role='seeker'`, then sign out + sign in with that account → confirm `useAuth().role === 'seeker'` and `/admin` shows "Access denied"
- Audit trail: this UPDATE will appear in `admin_audit_log` only if performed via the `admin_set_user_active` RPC route. Direct Studio SQL bypasses the audit chain. Acceptable here because it's a one-shot bootstrap operation, not a runtime admin action. Document the SQL + timestamp in the phase SUMMARY.

### Routing entry change in main.tsx

- Replace existing `path: '/admin'` route element from:
  ```tsx
  <ProtectedRoute requiredRole="admin">
    <AdminLayout>
      <DailyBriefing />
    </AdminLayout>
  </ProtectedRoute>
  ```
  to a new wrapper component (e.g., `<AdminGate>`) that renders the hybrid logic. AdminGate handles the loading/login/access-denied/admin branches; when admin, it composes `<AdminLayout><DailyBriefing /></AdminLayout>`.
- Sub-routes (`/admin/employers`, `/admin/seekers`, `/admin/jobs`, `/admin/placements`) keep their existing `<ProtectedRoute requiredRole="admin">` wrapping for now — they'll redirect to `/admin` (which now hosts AdminLoginPage for unauthenticated users) via `dashboardPathFor` if a non-admin lands there.

### Login.tsx admin-redirect compatibility

- The existing Login.tsx ternary fix (commit `0e91ff2`) routes admin-role JWTs from `/login` to `/admin` after sign-in. Phase 20.1 keeps that behavior — refactored to use `dashboardPathFor` (now imports the helper instead of inlining the ternary). User experience: admin can sign in via either `/login` or `/admin` and lands at `/admin` either way.

- **AUTH-FIX-02 spinner branch** (`ProtectedRoute.tsx:41-55`) — duplicate inline for the role-loading-timeout race window.

### Established Patterns

- **Single AuthContext subscription** — one `onAuthStateChange` for the whole app; AdminLoginPage MUST NOT add another subscription. Use `useAuth()` only.
- **Studio SQL for one-off DB writes** — CLAUDE.md §2; never `apply_migration` via MCP for runtime data changes (account bootstrap, role flips).
- **Atomic commits per task** — CLAUDE.md §4; each plan in 20.1 produces one commit per task.
- **TDD for refactor work** — Phase 14 + Phase 20 precedent (write tests first when changing behavior across multiple callsites).
- **Inline error rendering, not toast** — AuthLayout pattern uses sonner toasts but newer v2 surfaces favor StatusBanner inline. AdminLoginPage uses StatusBanner.

### Integration Points

- `src/main.tsx` route registry — `/admin` index route entry changes; `/admin/employers|seekers|jobs|placements` sub-routes unchanged
- `AuthContext.tsx` — no changes; helper imports from existing exports only
- `src/lib/routing.ts` — NEW file; clean integration point with no circular dependency risk
