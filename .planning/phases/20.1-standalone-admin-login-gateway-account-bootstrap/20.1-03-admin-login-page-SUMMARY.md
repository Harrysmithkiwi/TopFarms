## Decisions Made

1. **AdminGate branching order is LOAD-BEARING.** The `role === null` spinner check MUST come BEFORE the `role !== 'admin'` check, otherwise admin users in the AUTH-FIX-02 3 s race window would briefly see AccessDenied (RESEARCH Pitfall 1).
2. **Inline `role="alert"` div for both submit-error AND AccessDenied banner.** RESEARCH Pitfall 3: StatusBanner's variant union is fixed enum (`'shortlisted' | 'interview' | 'offer' | 'declined'`) with no `'error'` variant. The planner-recommended Option A (inline div with `--color-danger` / `--color-danger-bg` tokens) was applied — same pattern Phase 20-05 ProfileDrawer used.
3. **AdminLoginPage onSubmit does NOT navigate manually after `signIn()`.** RESEARCH Pitfall 9: explicit `navigate('/admin')` would race AuthContext role-loading. Instead, on success the function returns; AdminGate re-renders via AuthContext state-change and routes to the right branch. Login.tsx uses a `useEffect + didSubmit ref` workaround precisely because it lives at `/login`, not at the destination — AdminLoginPage avoids the workaround entirely by being co-located with the destination.
4. **`DailyBriefing` import dropped from main.tsx.** Now imported transitively via `AdminGate`. TypeScript was clean both with and without removal, but the plan instructs cleanup to avoid bundler dead-import warnings.
5. **Single atomic commit per CLAUDE.md §4.** The plan body explicitly directs one commit for AdminLoginPage.tsx + main.tsx + admin-login.test.tsx — the three files form one indivisible wave-3 hybrid-gate landing. (Per-TDD-step commits — RED test then GREEN code — were collapsed into the single ship per the plan's own instruction.)
