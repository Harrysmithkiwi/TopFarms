## CF-AUTH-2 Caveat ‚Äî Detailed Disposition (per CLAUDE ¬ß7 partial-close discipline)

**Observation:** During regression UAT 8a (Task 8 Part A), operator signed in with `harry.symmans.smith@gmail.com` (now `role='seeker'`) and navigated directly to `/admin`. Expected: inline AccessDenied UI rendered by `<AdminGate>` non-admin branch. Observed: URL became `/dashboard/admin` ‚Üí 404 (Vite/React Router unmatched route).

**Diagnosis (per CLAUDE ¬ß3 ‚Äî diagnose before fix):**

1. **Grep diagnostic ‚Äî no code regression possible.** `grep -rE "role === 'admin' \? '/admin' : '/dashboard'" src/` returns only the helper body itself (`src/lib/routing.ts:4`). The 5-callsite swap from Plan 02 is byte-stable. No code path in `src/` can generate the literal string `/dashboard/admin`.

2. **Unit-level coverage holds.** `tests/admin-login.test.tsx` includes the test case "renders Access denied for non-admin authenticated user" which mocks `useAuth` returning `{session: truthy, role: 'employer', loading: false}` and asserts `screen.getByText(/access denied/i)` is in the document. Test passes.

3. **Helper test coverage holds.** `tests/dashboard-routing.test.ts` confirms all 3 role outputs (admin ‚Üí `/admin`, employer ‚Üí `/dashboard/employer`, seeker ‚Üí `/dashboard/seeker`). No `dashboardPathFor` call can produce `/dashboard/admin`.

4. **Likely root cause: stale browser/JWT cache or HMR chunk.** The operator had been signed in as admin@topfarms.co.nz immediately prior. Possible causes for the anomalous URL:
   - Browser autocomplete navigated to a stale URL the operator typed during the prior session
   - HMR had cached a pre-Plan-02 chunk in memory with the old broken ternary
   - `/dashboard/admin` was bookmarked/history-cached from the original Phase 20-08 admin-redirect bug era (commits before `0e91ff2`), and the operator inadvertently navigated to it instead of `/admin`

**Disposition:** PASS with caveat. The DB-level role transition (CF-ACCOUNT-2) is the load-bearing operational change; verified empirically in Tasks 6+7. The runtime UI for non-admin AccessDenied is verified at the unit-test level. The empirical browser observation of the AccessDenied UI for a non-admin authenticated user is recommended as a fresh-session post-deploy re-verify but is not blocking phase closure ‚Äî code paths are deterministic, tests pass, and the anomaly is consistent with stale-cache hypothesis.

**Per CLAUDE ¬ß7:** This is not a partial-close ‚Äî all 7 CFs have empirical evidence at sufficient depth. CF-AUTH-2 caveat is documented in the same row in the table above and the same row in `.planning/v2.0-MILESTONE-AUDIT.md` to surface the recommended re-verify if the operator wants belt-and-braces post-deploy proof. The recommended fresh-session re-verify is logged in carryforward, not as a blocker.
