<!-- Recipient filter from notify-job-filled — used to determine which applicants SHOULD receive the email -->
NOTIFY_STATUSES (from supabase/functions/notify-job-filled/index.ts and supabase/migrations/017_notify_job_filled_webhook.sql):
  Inclusive: 'applied', 'review', 'interview', 'shortlisted', 'offered'
  Exclusive (terminal — no email): 'hired', 'declined', 'withdrawn'

Race-condition fix from Phase 13-01:
  In `MarkFilledModal.handleConfirm()`, the hired application's status is set to 'hired' BEFORE the job's status is set to 'filled', so the trigger sees the hired applicant already excluded from NOTIFY_STATUSES.

Trigger contract (from 017_notify_job_filled_webhook.sql):
  Trigger: `on_job_filled` fires when `jobs.status` transitions to `filled`.
  Function: `public.handle_job_filled()` calls `net.http_post(url, headers, body)` to `<supabase_url>/functions/v1/notify-job-filled` with `{ job_id }` body and bearer service_role_key from vault.

pg_net response inspection (from RESEARCH.md §Code Examples):
  `SELECT id, status_code, content_type, content, created FROM net._http_response ORDER BY created DESC LIMIT 10;`
  PASS: status_code = 200, content references {"sent": N, "failed": M, "job_id": "..."}
  FAIL signals: 404 (function not deployed) | 401 (auth wrong) | 5xx (function errored)

               COUNT(*) FILTER (WHERE a.status IN ('applied','review','interview','shortlisted','offered')) AS notify_count,
