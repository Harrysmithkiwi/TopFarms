CREATE POLICY "applications: employers update status for own jobs"
  ON public.applications FOR UPDATE
  USING (
    job_id IN (
      SELECT j.id FROM public.jobs j
      JOIN public.employer_profiles ep ON ep.id = j.employer_id
      WHERE ep.user_id = auth.uid()
    )
  );

CREATE POLICY "applications: seekers update own"
  ON public.applications FOR UPDATE
  USING (
    seeker_id IN (SELECT id FROM public.seeker_profiles WHERE user_id = auth.uid())

    seeker_id IN (SELECT id FROM public.seeker_profiles WHERE user_id = auth.uid())
  );

**6. compute_match_score(p_seeker_id uuid, p_job_id uuid) SQL function:**
`RETURNS jsonb LANGUAGE plpgsql SECURITY DEFINER`

Implementation logic:
- Load seeker from `seeker_profiles` WHERE id = p_seeker_id
- Load job from `jobs` WHERE id = p_job_id
- **Shed type (25pts):** Compare seeker `shed_types_experienced` with job `shed_type` (both text[]). Any exact overlap = 25pts. If seeker has 'rotary' and job has 'herringbone' (or vice versa) = 10pts (40%). No overlap = 0.
- **Location (20pts):** Same region = 20pts. Seeker `open_to_relocate` = true and different region = 16pts (80%). Adjacent region (use `get_adjacent_regions`) = 12pts (60%). Otherwise = 0.
- **Accommodation (20pts):** If seeker `accommodation_needed` = false → 20pts (no conflict). If true and job `accommodation` jsonb has `available = true` → start at 10pts, then +2.5 each for pets match (seeker pets not null AND job accommodation->>'pets' = 'true'), couples match, family match, utilities match. Cap at 20.
- **Skills (20pts):** Count seeker's skills (from seeker_skills) that match job's required skills (from job_skills). `skills_score = LEAST(20, ROUND(20 * matched_required / GREATEST(total_required, 1)))`. Give 60% weight for preferred skill matches.
- **Salary (10pts):** If job `salary_max` >= seeker `min_salary` → 10pts. If job `salary_max` is NULL → 5pts. If seeker `min_salary` is NULL → 10pts. If job `salary_max` < seeker `min_salary` → partial: `GREATEST(0, 10 - ROUND(10 * (seeker.min_salary - job.salary_max) / seeker.min_salary))`.
- **Visa (5pts):** If seeker `visa_status` IN ('nz_citizen', 'permanent_resident') → 5pts always. If seeker `visa_status` = 'working_holiday' AND job `visa_sponsorship` = true → 5pts. If seeker `visa_status` = 'needs_sponsorship' AND job `visa_sponsorship` = true → 5pts. Otherwise → 0.
- **Couples bonus (+5pts):** seeker `couples_seeking` = true AND job `couples_welcome` = true → +5.
- **Recency (x1.1):** If job `created_at` > now() - interval '7 days' → multiply total by 1.1, cap at 100.
- Return: `jsonb_build_object('total_score', v_total, 'breakdown', jsonb_build_object('shed_type', v_shed, 'location', v_location, 'accommodation', v_accommodation, 'skills', v_skills, 'salary', v_salary, 'visa', v_visa, 'couples', v_couples))`.

**7. compute_match_scores_batch(p_seeker_id uuid, p_job_ids uuid[]) SQL function:**
`RETURNS TABLE(job_id uuid, total_score int, breakdown jsonb) LANGUAGE plpgsql SECURITY DEFINER`
Loops over `p_job_ids` calling `compute_match_score` for each, returning results. This avoids N+1 on the search page.
