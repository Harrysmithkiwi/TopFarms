    2. Create `supabase/migrations/002_rls_policies.sql`:

       a. Create `get_user_role(user_id uuid)` security definer function that reads from user_roles
       b. Create `handle_new_user()` trigger function (security definer, plpgsql) that inserts into user_roles from `NEW.raw_user_meta_data->>'role'` with COALESCE default 'seeker'

       RLS policies for each table:
       - user_roles: users can read own role. No direct insert/update from client (trigger handles creation).
       - employer_profiles: employers CRUD own records (check role via get_user_role). Seekers can SELECT employer profiles (for viewing farm details on job listings).
       - seeker_profiles: seekers CRUD own records. Employers can SELECT seeker profiles where open_to_work = true (non-contact fields only — contact fields are in seeker_contacts).
       - seeker_contacts: seekers CRUD own records. Employers can SELECT only when a placement_fees record exists with acknowledged_at IS NOT NULL for that employer + seeker pair (via applications -> jobs -> employer_profiles join).
       - skills: all authenticated users can SELECT (read-only reference table)
       - jobs: employers CRUD own jobs. All authenticated users SELECT active jobs. Public (anon) can SELECT active jobs.
       - job_skills: employers CRUD for own jobs. All authenticated SELECT.
       - seeker_skills: seekers CRUD own. Employers SELECT.
       - match_scores: seekers SELECT own scores. Employers SELECT scores for applicants to their jobs.
       - applications: seekers INSERT and SELECT own. Employers SELECT applications to their jobs.
       - listing_fees: employers SELECT own. Insert via service role only (Stripe webhook).
       - placement_fees: employers SELECT own. Insert/update via service role only.
       - message_threads: basic self-access policies (Growth Phase — minimal policies)
