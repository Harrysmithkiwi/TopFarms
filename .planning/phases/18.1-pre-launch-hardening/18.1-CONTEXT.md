### Item #4 / orig #17 — `MarkFilledModal` atomicity via server-side RPC

- **Pattern: new SECURITY DEFINER RPC `mark_job_filled(p_job_id uuid, p_applicant_id uuid)`.** Both UPDATEs run inside the RPC body, transactional by construction. Client makes a single `supabase.rpc('mark_job_filled', { p_job_id, p_applicant_id })` call.
- Mirrors the `set_user_role` precedent in `supabase/migrations/018_set_user_role_rpc.sql` (SECURITY DEFINER + auth.uid() ownership check + idempotent semantics).
- Acceptance: on `UPDATE jobs` failure, the entire transaction rolls back — no orphan-hired application possible.
- The existing `notify-job-filled` trigger fires inside the transaction too (RPC's `UPDATE jobs` is what the trigger is bound to). Net behaviour for downstream subscribers is unchanged.
- `p_applicant_id` parameter is nullable — if NULL, the RPC only marks the job filled (no application UPDATE). Mirrors the current "no applicant selected" branch at `MarkFilledModal.tsx:70`.
- Auth check inside RPC: the calling employer must own the job (`jobs.employer_id` traces back to `auth.uid()` via `employer_profiles`). Match the access pattern used in `get-applicant-document-url`.
- Client side at `MarkFilledModal.tsx:62-101`: replace the 2 PostgREST `.update()` calls with one `.rpc()` call. Error-handling is single-branch.

### Item #5 / orig #19 — `match_scores` cleanup via DB trigger

- **Pattern: AFTER UPDATE trigger on `jobs` deletes orphaned `match_scores` rows when `OLD.status = 'active' AND NEW.status != 'active'`.**
- Trigger function: `cleanup_match_scores_on_status_change()` — `DELETE FROM match_scores WHERE job_id = NEW.id` (only fires when the OLD/NEW guard matches).
- One-time backfill in the same migration: `DELETE FROM match_scores ms WHERE EXISTS (SELECT 1 FROM jobs j WHERE j.id = ms.job_id AND j.status != 'active');` — clears the stale rows that accumulated pre-trigger.
- After this ships, the `!inner` PostgREST workaround in `SeekerStep7Complete.tsx` (commit `7401116`) becomes redundant but stays for defence-in-depth — removal is a 18.2 polish item, not 18.1.
- Failure mode: trigger error rolls back the `jobs.status` UPDATE. Acceptable — we'd want to know if cleanup is broken before status transitions silently leave debris.
- Reactivation case (job goes `archived` → `active`): match_scores get recomputed on demand by existing engine; no special handling needed in the trigger.

### Item #6 / orig #10a — 15× `unindexed_foreign_keys` advisor lints

- **Pattern: single migration `025_phase_18_1_fk_indexes.sql` adding 15 indexes**, one per FK identified by Supabase advisor. Tables: `admin_notes` (admin_id), `job_skills` (skill_id), `listing_fees` (employer_id, job_id), `message_threads` (employer_id, job_id, seeker_id), `messages` (sender_id, thread_id), `placement_fees` (application_id, employer_id, job_id, seeker_id), `saved_jobs` (job_id), `seeker_skills` (skill_id).
- Use `CREATE INDEX IF NOT EXISTS` on every line; idempotent.
- Apply via Supabase Studio SQL Editor (CLAUDE §2). Verify post-apply via read-only MCP `pg_indexes` query.
- Non-blocking on production: `CREATE INDEX` is online-safe; no reads/writes to these tables are blocked at runtime. Consider `CREATE INDEX CONCURRENTLY` only if the migration ends up running against a busy table — planner discretion based on table sizes (most are small placement/admin/messaging tables).
- The 4 NEW lint categories surfaced post-ROADMAP (`auth_rls_initplan: 37 was 33`, `multiple_permissive_policies: 61 NEW`, `unused_index: 8 NEW`) all fold into Phase 18.2's lint sweep — note added to ROADMAP §"Phase 18.2" successor commit. NOT in 18.1 scope.

### Item #3 / orig #15 — `X-Webhook-Secret` defence-in-depth (operator + code)

- **Pattern: shared secret stored as Supabase secret `WEBHOOK_SECRET` (matches `RESEND_API_KEY` precedent — `Deno.env.get('WEBHOOK_SECRET')`).** Generated once via `openssl rand -hex 32` (or equivalent).
- Two consumer Edge fns gain validation:
  - `notify-job-filled/index.ts` — read `req.headers.get('x-webhook-secret')` at handler entry; reject with 401 if missing or mismatched. Currently has zero validation.
  - `send-followup-emails/index.ts` — same shape.
- Two callers must include the header in their payloads:
  - `017_notify_job_filled_webhook.sql` — the `handle_job_filled` trigger uses `pg_net.http_post()` to call the function; the call needs to include the `X-Webhook-Secret` header. Header is read from a DB-level setting (`current_setting('app.webhook_secret')` or similar) so the SQL doesn't have the literal value baked in. Operator sets the DB-level config alongside the Supabase secret.
  - `011_phase_5_revenue.sql` (or whichever migration owns the `send-followup-emails` cron call) — same pattern.
- A constant-time comparison should be used inside the Edge fn (`crypto.timingSafeEqual` from Deno std OR a manual constant-time comparator) to avoid timing-attack leakage. Planner verifies what's available in Supabase Edge Function runtime.
- Cross-cutting test: vitest static-source test that asserts both Edge fns contain a `WEBHOOK_SECRET` env-var read AND a 401-on-mismatch branch. Mirrors the JOBS-01 regression-guard pattern.
- Existing TODO in `supabase/functions/get-resend-stats/index.ts:11-15` references this exact item — that fn is OUT of scope for 18.1 (it's a polling poller, not in the launch path); but the TODO should be cross-referenced in the plan so 18.2 picks it up.
- See CLAUDE §5 for context on why JWT-based gateway auth was disabled for these fns (`verify_jwt: false`); the X-Webhook-Secret check restores the defence-in-depth that was lost.
