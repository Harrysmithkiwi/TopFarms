|---|---|---|

**Why this is a CI re-enable blocker:**

Per Supabase CLI v2.95.4 source (`migration repair`):
`GetMigrationFile()` globs `migrations/<version>_*.sql`. The CLI expects the
registry version to match the filename prefix. The timestamp versions don't
match any on-disk file, so `db push --linked` will error with "Remote
migration versions not found in local migrations directory."

The existing supabase-deploy.yml comment (lines 4-16 pre-2026-05-21)
claims `supabase migration repair --status applied 20260428043338 ...`
fixes this — but per CLI source, that command requires a local file with
prefix `20260428043338` to exist, which it doesn't. The comment is wrong;
the repair would error.

**Mitigation applied (this commit):**

None — drift remains. Mitigated indirectly by gating migrations CI to
workflow_dispatch only.

**Resolution required (dedicated pre-flight phase before re-enable):**

In Supabase Studio SQL Editor (bypasses pooler):
```sql
BEGIN;
-- Remove timestamp-style rows (drift)
DELETE FROM supabase_migrations.schema_migrations

  '20260428043338', '20260428053314', '20260429031148',
  '20260503000000', '20260503210000'
);
-- Insert bare-sequence-prefix rows matching on-disk filenames
INSERT INTO supabase_migrations.schema_migrations
  (version, name, statements, created_by, idempotency_key, rollback)
VALUES
  ('018', 'set_user_role_rpc',                NULL, NULL, NULL, NULL),
```
