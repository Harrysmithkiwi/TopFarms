TopFarms

A New Zealand agriculture jobs marketplace connecting farm employers with skilled workers — replacing informal Facebook-group hiring with agriculture-specific matching (shed type, on-farm accommodation, DairyNZ qualifications, herd-size experience).

v2.0 “Launch Readiness” — functionally complete & launch-ready 1 blocker: PEND-01 Stripe live-mode swap

1 Overview & Status

What TopFarms is, where it stands, and the single thing blocking milestone close.

Core value

Farm employers and seekers find each other through agriculture-specific matching that no generic platform provides — shed type, accommodation sub-scoring, DairyNZ qualifications, and herd-size experience — delivering match quality that justifies switching from Facebook groups. The MVP targets the two sectors that dominate NZ ag hiring: dairy cattle and sheep & beef.

  • 86% of NZ ag Facebook job posts are dairy — the primary target sector.
  • 76% of NZ dairy seekers require on-farm accommodation — a major decision factor and scoring axis.
  • Shed type (rotary vs herringbone) is the primary skill differentiator in dairy.
  • 28% of seekers are couples — couples matching is a significant filter.

Milestone status

The product shipped v1.0 MVP (Phases 1–6, 2026-03-17) and v1.1 SPEC Compliance (Phases 7–11, 2026-03-23), and is now in v2.0 Launch Readiness — closing the launch-critical gaps so TopFarms can go live. Phase 22 (Pre-Launch P0 Closure) just completed. All 11 v2.0 requirements plus 19 internal Phase 21 ops requirement IDs are mapped to phases (0 unmapped) and empirically closed.

3 Architecture

Overall pattern: SPA + BaaS (Backend-as-a-Service) with role-based access control. A Vite + React Router v7 SPA delegates all backend operations to Supabase; authorization and persistence are server-enforced through Row-Level Security (RLS). Privileged operations run on Edge Functions with SECURITY DEFINER stored procedures.

Layers

  • Presentation — React components in src/components/ + src/pages/; layout wrappers (DashboardLayout, AdminLayout, JobSearchLayout) and UI primitives.
  • State management — src/contexts/AuthContext.tsx holds session, role, and isActive via a single app-wide onAuthStateChange subscription (defence-in-depth against multi-tab Web Lock contention, AUTH-FIX-02).
  • Access control (frontend) — src/components/layout/ProtectedRoute.tsx guards session, role, and active status with a requiredRole prop.
  • API layer — src/lib/supabase.ts (createClient() with anon key; persistent sessions + auto-refresh).
  • Database — supabase/migrations/; 33 migrations defining 14+ tables, all with RLS enabled.
  • Privileged ops — 11 Edge Functions in supabase/functions/ + admin RPCs in migration 023 (SECURITY DEFINER + GRANT EXECUTE TO authenticated).

Three-tier role model

RoleCapabilities
employerPost listings, view applicants, mint document URLs, manage placement fees
seekerUpload documents, apply for jobs, save searches
adminDocument review queue, employer/seeker management, suspension toggle (Phase 21)

Roles live in the user_roles table with an is_active flag, checked in RLS policies and Edge Functions.

Key tables

user_roles, employer_profiles, seeker_profiles, jobs, applications, seeker_documents, seeker_contacts (separate table for contact masking), saved_searches, admin_audit_log, and admin_notes. All tables have ENABLE ROW LEVEL SECURITY; policies check auth.uid() and role via the get_user_role(p_user_id) SECURITY DEFINER helper (read-once to avoid recursion).

Key abstractions & patterns

RLS policy template

CREATE POLICY "employer_profiles: employers manage own"
  ON public.employer_profiles
  FOR ALL
  USING (user_id = auth.uid() AND get_user_role(auth.uid()) = 'employer')
  WITH CHECK (user_id = auth.uid() AND get_user_role(auth.uid()) = 'employer');

Admin RPC pattern (migration 023)

Every admin RPC opens with an admin gate, writes an audit row before returning, and is callable only by authenticated users (the gate enforces the role):

CREATE OR REPLACE FUNCTION public.admin_set_user_active(...)
RETURNS jsonb LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$
BEGIN
  PERFORM public._admin_gate();   -- raises if role != admin
  -- mutation + INSERT INTO admin_audit_log ...
END; $$;
GRANT EXECUTE ON FUNCTION public.admin_set_user_active TO authenticated;

Gateway-trust JWT pattern (CLAUDE.md §5)

For Edge Functions with verify_jwt: true, the Supabase gateway has already validated the JWT signature upstream. Functions must not re-validate via adminClient.auth.getUser(token) (that path rejects valid ES256 tokens — BFIX-05). Instead, trust the gateway and decode locally: 

const payload = JSON.parse(atob(token.split('.')[1].replace(/-/g, '+').replace(/_/g, '/')))
if (payload.aud !== 'authenticated') return 401
const callerUserId = payload.sub

Reference implementation: supabase/functions/get-applicant-document-url/index.ts:79-98, which adds a 5-layer auth check (method, JWT, role, relationship, document-type whitelist).

Representative data flows

  • Seeker signup → onboarding: supabase.auth.signUp() with role: 'seeker' → on_auth_user_created trigger creates the user_roles row → verification → /onboarding/seeker wizard upserts profile data step-by-step under RLS.
  • Employer views applications: get_applicants_for_job RPC (migration 030) validates ownership → employer invokes get-applicant-document-url Edge Function (5-layer auth) → 15-minute signed URL.
  • Admin document verification (Phase 21): admin_list_document_queue → approve/reject RPC fires _admin_gate(), writes audit log, then calls send-document-status-email via pg_net.http_post().

Entry points

  • src/main.tsx — createBrowserRouter with 30+ routes, wraps the app in AuthProvider, mounts the Sonner <Toaster />.

6 Conventions

Codified patterns the codebase follows — forms, data access, error feedback, and commit discipline.

Imports & aliases

Order: React/framework → third-party UI/util → internal @/ imports → relative (rare). Always use the @/* → ./src/* alias for internal imports.

Forms & validation

Zod schemas defined at module level; TypeScript types inferred via z.infer<typeof schema>; bridged to react-hook-form with zodResolver. Zod error messages are user-facing.

const schema = z.object({
  title: z.string().min(3, 'Title must be at least 3 characters'),
  contract_type: z.enum(['permanent', 'contract', 'casual']),
})
type FormData = z.infer<typeof schema>
const { register, handleSubmit, formState: { errors } } =
  useForm<FormData>({ resolver: zodResolver(schema) })

Supabase access

  • Import the singleton: import { supabase } from '@/lib/supabase'.
  • Reads: .from('table').select().eq(...).single() / .maybeSingle(). Protected ops: .rpc('fn', { p_arg }) against SECURITY DEFINER functions.
  • Always destructure and check { data, error }; return sensible defaults on error (e.g. loadRole returns isActive: true on DB failure so a transient error never wrongly suspends a user).
  • Race long auth operations against a 3s timeout via Promise.race() (loadRoleWithTimeout pattern).

Error feedback (Sonner toasts)

Persistent errors (auth/validation failures requiring action) must use toast.error(msg, { duration: Infinity, closeButton: true }) — the SIGNUP-01 precedent. A default 4s auto-dismiss caused the original SIGNUP-01 bug where users never saw the email_address_invalid error. Transient system messages may use the default duration.

Style & typing

  • Prettier (semicolons, trailing commas, 2-space indent). ESLint v9 flat config.
  • TypeScript strict mode: strict, noUnusedLocals, noUnusedParameters, noFallthroughCasesInSwitch, noUncheckedSideEffectImports all on.
  • Booleans prefixed is/has; module-level constants UPPER_SNAKE_CASE; props interfaces suffixed Props.
  • Outcome types use discriminated unions, e.g. type LoadRoleOutcome = { ok: true; ... } | { ok: false; reason: 'timeout' }.
  • Tailwind v4 utility classes; merge with clsx() / cn(); avoid inline styles except branded OAuth buttons.

Comments & commits

  • Prefer self-documenting code; band-aid code is tagged with a ticket ID prefix, e.g. // [AUTH-FIX-02] — never an untracked TODO.
  • Atomic commits (CLAUDE.md §4): one phase/sub-phase per commit. Format type(scope): description; body explains why; every commit carries the Co-Authored-By: Claude trailer.

Signature patterns

1. vi.hoisted() for statically-imported mocks (critical)

When the SUT statically imports a module (e.g. @/lib/supabase or sonner), the mock must be hoisted so it installs before compile-time resolution:

const { toastErrorMock } = vi.hoisted(() => ({ toastErrorMock: vi.fn() }))
vi.mock('sonner', () => ({
  toast: { error: toastErrorMock, success: vi.fn() },
  Toaster: () => null,
}))
// static import of SUT comes AFTER the hoisted mock
import { SignUp } from '@/pages/auth/SignUp'

2. Chainable Supabase query mocks

Mock the full .from().select().eq().order().limit() chain, terminating in a resolved { data, error }:

fromMock.mockReturnValue({
  select: vi.fn().mockReturnValue({
    eq: vi.fn().mockReturnValue({
      order: vi.fn().mockReturnValue({
        limit: vi.fn().mockResolvedValue({ data: rows, error: null }),
      }),
    }),
  }),
})

3. The static-source-guard pattern

A regression net that readFileSync()s a source file and asserts code shape with regex — catching type-level bugs that are silent at runtime (e.g. HOMEBUG-02, where a string value was passed to an int column). Includes both positive assertions (the fix must exist) and negative assertions (the pre-fix shape must not). Very fast (<5ms), no DOM, no network.

4. The it.todo deferral pattern

Tests are scaffolded as it.todo('...') stubs documenting intent before implementation; they show as “todo” (not failing) and are promoted to real it() blocks when the feature ships. This accounts for the ~114 deferred tests.

Conventions

  • Multiplexed RPC mocks switch on function name inside rpcMock.mockImplementation.
  • Async assertions use waitFor() / findBy*; state updates wrapped in act().
  • Layout chrome (e.g. DashboardLayout) is passthrough-mocked to avoid heavy provider trees.
  • Every test file opens with a header comment: phase + feature ID, what flips RED/GREEN, and the mock strategy.
  • Coverage isn't enforced in config; passing-count + pass-rate is the primary signal.

Security workarounds (legacy HS256 JWT limitation)

X-Webhook-Secret defence-in-depthMEDIUMnotify-job-filled + send-followup-emails

Both functions run with verify_jwt: false because the gateway uses legacy HS256 signing for service-role JWTs and rejects them when verify_jwt: true (the BFIX-05 HS256/ES256 asymmetry). Instead they validate a shared X-Webhook-Secret header, injected by the on_job_filled trigger from vault.decrypted_secrets.webhook_secret (migration 029). Weaker than asymmetric signing and adds rotation overhead.

Plan for removal (Item #16, dedicated phase): migrate the project JWT signing from legacy HS256 to ES256/JWKS, then flip both functions to verify_jwt: true, remove the header validation/injection chain, and decommission the vault secret. Wide blast radius (all service-role usage).

Gateway-trust JWT pattern (verify_jwt=true fns)PATTERNprevents regressions

Not a bug — a critical pattern. With verify_jwt: true, the gateway validates the signature upstream; functions must decode the JWT payload locally (sub, aud === 'authenticated') rather than calling adminClient.auth.getUser(token), which rejects valid ES256 tokens. Codified in CLAUDE.md §5; correctly used in get-applicant-document-url/index.ts:79-98 and send-document-status-email. Any new function copying the rejected pattern breaks identically — guarded by a static-source regression test.

Security hardening gaps

CORS-01 — wildcard Allow-OriginLOW

Privacy-critical Edge Functions return Access-Control-Allow-Origin: '*'. Safe today because authorization (role, relationship, identity exclusion) is enforced in the handler — the CORS header doesn't grant access. Post-launch: scope to the stable production domain + localhost:5173.

PRIV-01 — 404 vs 403 existence leakLOW

In get-applicant-document-url/index.ts, an existing-but-unowned UUID returns 404 (“Application not found”) and an unowned-job UUID returns 403, technically leaking existence. Low risk because application/document UUIDs are not enumerable. Post-launch: unify both to 403 “Document not accessible”.
