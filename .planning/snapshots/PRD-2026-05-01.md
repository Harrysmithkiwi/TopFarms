## 1. Product Overview

TopFarms is a New Zealand agricultural job marketplace connecting farm employers with skilled workers. It replaces informal hiring via Facebook groups with a structured platform featuring agriculture-specific filters (shed type, accommodation, DairyNZ qualifications), AI-powered match scoring, verified employer profiles, and full SPEC v3.0 compliant UI — purpose-built for the NZ ag sector.

### Core Value Proposition

Farm employers and seekers find each other through agriculture-specific matching no generic platform provides — shed type, accommodation sub-scoring, DairyNZ qualifications, herd size experience.

### Market Context

- NZ agricultural sector is small and reputation-driven — social enforcement informs platform design
- 86% of NZ ag Facebook job posts are dairy — primary target sector
- 76% of NZ dairy seekers require on-farm accommodation
- Shed type (rotary vs herringbone) is the primary skill differentiator in dairy
- 28% of seekers are couples — couples matching is a significant filter

## 9. Key Architecture Decisions

### v1.0 + v1.1 Decisions

Unchanged from `PRD.md` §9 — pre-computed match scores, RLS-layer contact masking, `seeker_contacts` separate table, flat placement fee tiers ($200/$400/$800), Stripe Invoice Net 14, manual NZBN verification for MVP, hardcoded market-rate salary data, manual skills curation, Resend via plain `fetch()` not SDK, Tailwind v4 CSS-first, ChipSelector `string[]` value shape, private `seeker-documents` bucket, `estimate_match_pool` RPC with DEFAULT NULL params, AI candidate summary cache-first, motion library for animations.

### v2.0 Decisions (worth preserving across context resets)

| Decision | Rationale |
|----------|-----------|
| **Gateway-trust JWT pattern for `verify_jwt: true` Edge Functions** | The Supabase gateway has already validated the JWT signature upstream when `verify_jwt: true`. Re-validating via `adminClient.auth.getUser(token)` on a service-role-keyed client routes `/auth/v1/user` differently and rejects valid ES256 tokens. Decode the JWT payload locally with `atob`, validate `aud === 'authenticated'`, take `sub`. Codified in `CLAUDE.md` §5. Reference: `supabase/functions/get-applicant-document-url/index.ts:75-94`. Established BFIX-05. |
| **Defence-in-depth on identity-document exclusion** | Identity documents are filtered at three independent layers (RLS migration 020 + Edge Function `EMPLOYER_VISIBLE_DOCUMENT_TYPES` whitelist + listing-query filter). A bypass requires defeating all three. Cross-layer drift risk is acknowledged tech debt for Phase 18. |
| **Atomic-commit discipline (one phase per commit)** | One phase or sub-phase per commit. Don't bundle unrelated work. Amending within the same atomic-commit-window is acceptable when internal consistency requires it (precedent: BFIX-02/03 + BFIX-01 docs reconciliation amended into Phase 14-03 commit `e8f0882`). Codified in `CLAUDE.md` §4. |
| **Diagnose before fix** | No silent edits, even on docs. Show SQL/code/diff before writing. Read logs and form a hypothesis before changing code — the cost of being wrong about the fix exceeds the cost of the diagnosis step. Precedent: BFIX-05 took three rounds of diagnosis before the gateway-trust pattern surfaced. Codified in `CLAUDE.md` §3. |
| **Project-scoped MCP only** | Project-scoped `.mcp.json` at repo root with project ref `inlagtgpynemhipnqvty`. Never rely on user-home `~/.mcp.json` — different machines/sessions resolve it differently. Verify project ref before any Supabase tool call. Codified in `CLAUDE.md` §1. |
| **`--read-only` MCP by default; Studio for one-off DB writes** | `/mcp Reconnect` does NOT respawn the MCP subprocess with new args — flag changes only propagate on a full Claude Code restart. Preferred path for one-off DB writes: Supabase Studio SQL Editor. Codified in `CLAUDE.md` §2. |
| **Dual-state migration registry (sequence-prefix on disk, timestamp in registry)** | Disk filenames use `NNN_short_name.sql` for human readability + `git diff` ordering. Registry uses CLI-generated `YYYYMMDDHHmmss` timestamps generated at apply-time. The two are intentionally decoupled. Lookup table in `supabase/migrations/NAMING.md`. Studio-applied migrations leave registry rows absent — runtime artefacts (`pg_extension`, `pg_proc`, `pg_trigger`) are the source of truth, not `list_migrations`. |
| **Migration registry repair via Studio SQL INSERT** | `supabase migration repair` CLI fails for this project due to filename-glob-vs-registry mismatch. Repair via direct `INSERT INTO supabase_migrations.schema_migrations` from Studio SQL Editor. Pattern documented in `supabase/migrations/NAMING.md` and established in plan 15-03. |
| **Partial-close discipline for REQUIREMENTS.md** | When a requirement has multiple gaps (e.g., deploy gap + runtime/secret gap + E2E proof gap), closing one gap is NOT the same as satisfying the requirement. Do not flip `[ ]` to `[x]` unless ALL gaps are empirically closed. Add carryforward entries to milestone audit for unclosed gaps. Established Phase 15 closeout (MAIL-01/02 kept open with partial-close notes despite deploy completion). |
| **AuthProvider centralisation** | Single `getSession` + single `onAuthStateChange` for the whole app, replaces 22 independent `useAuth` instances that contended the Supabase auth-token Web Lock and produced AbortError lock-stealing (JOBS-01). `TOKEN_REFRESHED` event explicitly skipped so cross-tab token sync doesn't reload role. 3s `loadRole` timeout retained as defence-in-depth. |
| **CI deploy via GitHub Actions, gated decisions captured** | `.github/workflows/supabase-deploy.yml` triggers on push to `main` matching `supabase/migrations/**` or `supabase/functions/**` paths. Sub-decisions captured in plan 15-03 evidence: trigger (push-to-main), scope (migrations + functions), permissions (`SUPABASE_ACCESS_TOKEN` PAT + `SUPABASE_DB_PASSWORD` GH Actions secrets), failure mode (notify-only on first iteration; rollback strategy deferred). |
